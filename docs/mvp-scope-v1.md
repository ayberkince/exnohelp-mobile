# MVP Scope v1

## Purpose
This document defines the first realistic launch version of the product.

It is designed to keep the launch disciplined and focused.

The goal of v1 is not to build every possible feature.
The goal of v1 is to make the marketplace work safely and clearly in a narrow form.

---

## 1. Launch Scope Summary

### Launch Model
- Berlin only
- web-first, responsive mobile-first design
- two-sided marketplace
- non-medical support only
- manual verification and moderation in early stages
- narrow service categories only

### User Roles
- Client
- Helper
- Admin / Operator

---

## 4. Not Included in v1

The following features are intentionally excluded from the first launch version:

### Medical / Clinical Features
- diagnosis tools
- medical advice engine
- AI clinical support
- prescription interpretation
- symptom checker
- nursing workflows
- treatment planning

### Marketplace Expansion Features
- multi-city support
- multiple countries
- full native iOS and Android apps
- subscriptions
- advanced referral systems
- video calls
- family accounts
- corporate accounts
- agency or institution dashboards

### Automation / Intelligence Features
- full automated moderation
- advanced recommendation engine
- AI trust scoring
- advanced fraud models
- automated emergency triage

### Convenience Features
- multilingual AI translation engine
- advanced scheduling automation
- route optimization
- group support requests
- recurring care subscriptions

---

## 5. Service Categories Included in v1
The marketplace will support only these categories at launch:

- accompaniment to appointment
- waiting support
- return-home accompaniment
- emotional companionship
- light paperwork support
- light practical help after appointment

---

## 6. Core Marketplace Logic in v1

### Client Discovery
A client can:
- browse helpers directly
- compare profiles
- review trust markers
- book a helper

### Helper Discovery
A helper can:
- browse open requests
- filter by category, time, location, language, and price
- accept or apply

This dual-discovery model is included in v1 because it improves marketplace liquidity.

---

## 7. Payment and Booking Logic in v1
The first version must support:
- client payment
- platform fee
- helper payout workflow
- cancellation states
- refund states
- booking completion state
- dispute or report state

---

## 8. MVP Success Criteria
v1 is successful if:
- clients can create requests and book helpers
- helpers can create profiles and accept work
- bookings can be completed end-to-end
- payments can be processed
- reviews can be left
- helpers can be verified manually
- unsafe or suspicious activity can be reported and reviewed
- the platform can operate in a controlled Berlin launch

---

## 9. Build Principle
Every feature added to v1 must answer one of these needs:
- increase trust
- make booking clearer
- improve marketplace liquidity
- reduce operational risk
- improve real user usability

If a feature does not serve one of these goals, it should not be included in v1.
