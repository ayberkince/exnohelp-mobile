# Refund Ops v1

## Purpose
This document defines the internal operational process for refunds and payout holds.

---

## Refund Triggers
- helper no-show
- severe issue before service completion
- cancellation under refund-eligible timing
- payment mistake
- admin trust/safety intervention

---

## Refund Review Checklist
Operator should check:
- booking status
- cancellation timing
- report/dispute status
- payment status
- payout status
- prior admin actions

---

## Possible Outcomes
- no refund
- partial refund
- full refund

---

## Payout Hold Cases
Payout may be held if:
- dispute open
- serious report pending
- fraud concern
- refund under review

---

## Logging Rule
Refunds and payout holds must always be logged with:
- booking reference
- reason
- operator
- timestamp
