# Booking Ops v1

## Purpose
This document defines how operators handle bookings that require review.

---

## Booking Cases Requiring Ops Review
- payment succeeded but booking state is broken
- booking canceled after payment
- no-show report
- severe lateness issue
- dispute after completion
- payout needs hold
- refund required
- booking linked to safety concern

---

## Booking Review Checklist
Operator should check:
- booking status
- payment status
- payout status
- report history
- messages
- user status
- timeline of events

---

## Possible Operator Actions
- no action
- update booking state
- contact users
- hold payout
- trigger refund review
- cancel booking
- escalate to trust and safety review

---

## Operational Rule
No financial action should happen without checking both booking state and payment state.
