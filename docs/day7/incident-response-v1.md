# Incident Response v1

## Purpose
This document defines how the platform handles incidents that occur around bookings.

An incident is any event that creates meaningful safety, reliability, or trust concern.

---

## Example Incidents
- helper no-show
- client no-show
- inappropriate in-person behavior
- prohibited-task request
- chat harassment escalation
- off-platform payment attempt
- suspected fraud
- strong dispute over completed service

---

## Incident Handling Flow
1. incident is reported or detected
2. incident record is created
3. severity is assessed
4. immediate operational action is chosen if needed
5. financial effects reviewed if relevant
6. user account actions reviewed if relevant
7. incident is resolved and logged

---

## Immediate Action Types
- no immediate action
- user warning
- booking cancellation
- payout hold
- temporary account pause
- escalation to full review

---

## Incident Logging Requirements
Each incident record should capture:
- users involved
- booking involved
- category
- description
- evidence if any
- current state
- action taken
- admin notes

---

## v1 Operational Rule
In v1, incident handling can be manual as long as:
- it is documented
- it is consistent
- account and payout actions are logged
