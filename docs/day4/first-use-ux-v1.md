# First Use UX v1

## Purpose
This document defines what a user sees and is encouraged to do in the first sessions.

The first-use experience must reduce confusion and lead users toward one meaningful action.

---

## 1. First Client Session

### Primary goal
Get the client to:
- understand the service boundaries
- trust the platform
- create a first request or browse helpers

### First client dashboard priorities
- welcoming hero text
- simple explanation of what support can be booked
- strong trust-and-safety reminder
- clear CTA to create first request
- secondary CTA to browse helpers

### Recommended first client CTA
Create your first support request

---

## 2. First Helper Session

### Primary goal
Get the helper to:
- complete a strong profile
- understand service boundaries
- start verification
- browse open requests

### First helper dashboard priorities
- profile completion indicator
- verification progress card
- list of open requests
- explanation of how bookings work
- trust and behavior standards reminder

### Recommended first helper CTA
Complete your profile and verification

---

## 3. Empty State UX
The app must have strong empty states.

### Client empty states
- no bookings yet
- no saved helpers yet
- no requests yet

### Helper empty states
- no completed bookings yet
- no active requests accepted yet
- no reviews yet

Each empty state should contain:
- reassurance
- next best action
- minimal friction CTA

---

## 4. Trust Messaging in First Use
Trust should be visible in the first user sessions through:
- safety banners
- clear prohibited-task messaging
- verification explanation
- in-app payment reminder
- reporting availability

---

## 5. Progressive Disclosure
Do not overload the first session with too much complexity.

The app should reveal complexity step by step:
- role
- profile
- trust
- action
- deeper tools later
